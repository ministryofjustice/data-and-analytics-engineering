# Security Model

This document describes the main security controls around the documentation site build and publishing pipeline.

## Dependency Installation

The project uses `npm ci` in CI to install exactly what is recorded in `package-lock.json`. This keeps builds reproducible and prevents the lockfile from being silently updated during deployment.

## Pinned GitHub Actions

The workflows in `.github/workflows/` pin third-party GitHub Actions to commit SHAs rather than mutable tags. This reduces the risk of an upstream tag being changed after review.

## SLSA Supply Chain Verification

Both deployment and preview workflows include the Ministry of Justice SLSA action. This adds supply-chain checks around dependency updates and build inputs before publishing.

## Minimal Workflow Permissions

The deploy workflow uses `contents: write` so it can publish to `gh-pages`. The preview workflow adds `pull-requests: write` so it can comment on pull requests and `pages: read` for preview publishing. Workflows do not use broad `write-all` permissions.

## GitHub Pages Publishing

The generated `_site/` directory is deployed to the `gh-pages` branch. The main deploy workflow preserves the `pr-preview/` directory so open pull request previews are not removed by main-site deployments.

## Pull Request Previews

PR previews are built with a `PATHPREFIX` of `/pr-preview/pr-{PR_NUMBER}/`. The site stylesheet is generated by `src/content/styles.11ty.js`, which rewrites the GOV.UK asset path for the current prefix so preview assets resolve correctly.

## Publisher Guidance Boundary

The root `docs/` folder is not published to the website. Public content should be added under `src/content/`; publisher-only guidance should stay in root `docs/`.

## Dependency Updates

Dependabot is configured in `.github/dependabot.yml` to monitor npm dependencies. Dependency update pull requests should be reviewed with the same preview workflow as content changes.
